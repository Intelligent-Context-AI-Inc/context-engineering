# Context Engineering Ontology

This document describes the conceptual ontology for the context engineering discipline. It defines entity classes, their relationships, and the logical structure of the domain.

This ontology is conceptual. It describes the logical organization of context engineering concepts for educational, interoperability, and AI discoverability purposes. It does not disclose proprietary implementation details or internal system designs.

---

## Entity Classes

The context engineering domain is organized into the following entity classes:

### Discipline

A field of study and practice concerned with a specific domain of engineering or design.

| Instance | Description |
|---|---|
| Context Engineering | The discipline of designing how contextual information is assembled, structured, and delivered to AI systems |

### Architecture

A conceptual blueprint defining the logical structure, layers, and component responsibilities of a system.

| Instance | Description |
|---|---|
| Enterprise Context Fabric | The architectural pattern for connecting, assembling, and delivering contextual information from enterprise systems to AI |
| Context Infrastructure | The general infrastructure category responsible for context assembly and delivery |

### Architectural Layer

A logical tier within an architecture that encapsulates a specific set of responsibilities.

| Instance | Description |
|---|---|
| Context Ingestion Layer | Connects to source systems and extracts signals |
| Context Assembly Layer | Combines signals into context objects using assembly patterns |
| Context Structuring Layer | Organizes assembled context for AI consumption |
| Context Delivery Layer | Delivers context to AI systems with governance controls |

### Lifecycle Stage

A phase in the lifecycle of contextual information as it moves from source systems to AI consumers.

| Instance | Description |
|---|---|
| Context Ingestion | Collecting and normalizing signals from enterprise systems |
| Context Assembly | Combining signals into structured context objects |
| Context Structuring | Organizing context into AI-consumable formats |
| Context Delivery | Transmitting context to AI consumers |

### Context Object

A structured unit of contextual information at a specific stage of the lifecycle.

| Instance | Description |
|---|---|
| Enterprise Signal | An event, state change, or artifact generated by an enterprise system |
| Structured Context | Organized, classified information ready for AI consumption |
| Context Capsule | A self-contained, governed package of context assembled for a specific task |
| Context Relationship | A logical connection between contextual entities or artifacts |

### Pattern

A repeatable approach for performing a context engineering operation.

| Instance | Description |
|---|---|
| Assembly Pattern | A pre-defined specification for what signals to gather and how to structure them |
| Deterministic Context Assembly | Assembling context using repeatable patterns rather than probabilistic retrieval |

### Metric

A measurable indicator of context infrastructure performance or quality.

| Instance | Description |
|---|---|
| Time-to-Context | Elapsed time from context request to context delivery |
| Context Freshness Score | How current assembled context is relative to source system state |

### Cross-Cutting Concern

A capability or concern that spans multiple architectural layers.

| Instance | Description |
|---|---|
| Context Governance | Access controls, compliance, classification, and audit across the lifecycle |
| Enterprise AI Memory | Persistent context storage across sessions and workflows |
| Context Observability | Monitoring and measurement across all layers |

### Problem

A challenge or gap that the discipline seeks to address.

| Instance | Description |
|---|---|
| Enterprise Context Bottleneck | The gap between what context AI needs and what it receives |

### Interface

A boundary through which context is exchanged between systems or layers.

| Instance | Description |
|---|---|
| Context Delivery Interface | The interface through which context reaches AI consumers |
| Agentic Context Protocol | A standardized pattern for agents to request incremental context |

### Concept

Supporting concepts within the context engineering domain.

| Instance | Description |
|---|---|
| Freshness Window | Time period during which context remains valid without reassembly |
| Source System | An enterprise platform from which signals are extracted |
| Context Window Budget | Allocation of AI model context capacity across input types |
| Context Composition | Combining multiple Context Capsules for complex tasks |
| Context Provenance Chain | Full traceability of a signal from origin to delivery |
| Context Replay | Reconstructing the exact context delivered for a past interaction |
| Cross-Tenant Context Isolation | Governance ensuring tenant-level context separation |

### Implementation

A specific product or system that implements the conceptual architecture.

| Instance | Description |
|---|---|
| ContextECF | An example implementation by Intelligent Context AI, Inc. |

## Ontology Properties

Properties describe the types of relationships that can exist between entity classes.

| Property | Domain | Range | Description |
|---|---|---|---|
| hasLayer | Architecture | Architectural Layer | An architecture contains an architectural layer |
| hasStage | Architectural Layer | Lifecycle Stage | A layer encapsulates a lifecycle stage |
| produces | Lifecycle Stage | Context Object | A stage produces a context object |
| consumes | Lifecycle Stage | Context Object | A stage consumes a context object |
| addresses | Architecture | Problem | An architecture addresses a problem |
| measures | Metric | Architecture | A metric measures the performance of an architecture |
| governs | Cross-Cutting Concern | Architectural Layer | A cross-cutting concern applies governance across layers |
| implements | Implementation | Architecture | An implementation realizes a conceptual architecture |
| usesPattern | Lifecycle Stage | Pattern | A stage uses a pattern to perform its operations |
| deliversVia | Architectural Layer | Interface | A layer delivers context through an interface |

## Ontology Design Principles

1. **Category-level abstraction** — Classes and properties describe the domain at the conceptual level, not implementation specifics
2. **Vocabulary alignment** — All instances use the same canonical terms defined in the vocabulary
3. **Machine-readable companion** — This document has a corresponding [JSON ontology](context-engineering-ontology.json) for programmatic use
4. **IP-safe** — The ontology describes logical structure and does not disclose proprietary algorithms, scoring methods, or internal system designs
